# 通訊架構設計外部參考

## 工作背景

- 任務：評估後端元件職責與通訊設計
- 議題：Elysia websocket 是否應再包 transport abstraction
- 議題：是否需要 EventBus
- 議題：如何安放在線中的玩家與對局關係

## 查詢方向

- Elysia websocket 官方能力與邊界
- Bun / Elysia pubsub 的適用範圍
- 遊戲伺服器常見的 transport / room / state 分層
- Event-driven backend 在遊戲情境的適用時機

## 關鍵發現

- Elysia `.ws()` 是對 Bun websocket 的薄封裝
- Elysia 提供 `send`、`publish`、`subscribe`、`unsubscribe`
- topic pubsub 比較像底層廣播能力，不等於應用層 session / room 模型
- 若直接在業務流程中依賴 `ws` 與 topic 字串，會讓遊戲語意散落在 adapter 層
- 類似 Colyseus 的遊戲伺服器仍明確分出 transport layer、room/session layer、game logic layer
- EventBus 在多模組共同訂閱事件時較有價值；單一路徑 command flow 未必要先引入

## 對本案可用結論

- 適合保留一層 `PlayerTransport` 或同等抽象，隔離 `ElysiaWS`
- `PlayerTransport` 不宜同時承擔在線玩家與對局關係管理
- 建議把在線狀態獨立成 registry 類元件
- 目前主流程可先維持 direct call
- 若未來有多個模組要對上下線、加入對局、結束對局反應，再補 EventBus 或 domain events
