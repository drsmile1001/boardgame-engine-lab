# 通訊架構設計

## 目標

- 在個人可維護的範圍內，建立後端元件職責與通訊分層
- 保留 websocket 長連線模型
- 避免業務邏輯直接依賴 Elysia / Bun 的 `ws` 物件
- 先以容易理解與容易迭代的設計為主

## 設計原則

- `ws` 物件只作為 adapter 層資源，不直接成為業務模型
- 核心流程先採 direct call
- 優先把元件責任切乾淨，再考慮 event bus 這類進一步抽象
- 單一玩家允許多連線，方便單機 e2e 測試與多分頁操作
- reconnect 暫不納入本輪設計

## 元件分層

### `GameRule`

- 遊戲規則
- 定義遊戲狀態轉移規則
- 提供純規則函式，例如 `move(playerId, action): GameState`
- 不依賴 websocket、repo、topic、Elysia 型別
- 應能獨立測試

### `Game`

- 特定一場遊戲的實例
- 包含遊戲狀態、玩家資訊、遊戲狀態欄位
- 目前程式中仍有 `Match` 命名
- `Match` 是否更名為 `Game` 另行評估，不在本輪強制調整

### `GameRepo`

- 遊戲存放庫
- 負責讀寫 `Game`
- 提供 `get(gameId): Game` 與 `save(gameId, game)` 類型的介面
- 保持「資料就是資料」的責任，不承擔即時通訊與連線生命週期

### `GameRunner`

- 遊戲執行器
- 作為應用層 orchestrator / command handler
- 接收玩家命令後
  - 載入 `Game`
  - 驗證玩家與遊戲上下文
  - 呼叫 `GameRule`
  - 保存新狀態
  - 要求 `PlayerTransport` 回送或廣播結果
- 不直接操作 `ws`

### `PlayerTransport`

- 玩家通訊抽象
- 原 `SessionTransport` 建議更名為 `PlayerTransport`
- 負責銜接 websocket 與應用層流程
- 提供玩家導向的送收能力，例如
  - `connect(connectionContext)`
  - `disconnect(connectionId)`
  - `receiveMessage(connectionId, message)`
  - `sendToPlayer(playerId, message)`
  - `broadcastToGame(gameId, message)`
- 負責把 websocket 訊息轉交給 `GameRunner`
- 不直接承擔完整的在線遊戲關係管理

### `OnlineSessionRegistry`

- 在線狀態存放庫
- 專責管理記憶體中的在線關係
- 建議管理的資料包含
  - `connectionId -> playerId`
  - `playerId -> connection[]`
  - `playerId -> gameId`
- 因允許單一玩家多連線，資料結構應以多 connection 建模
- 暫不處理 reconnect 所需的額外恢復資訊

### `PlayerSocket`

- Elysia websocket adapter
- 只負責對接 websocket lifecycle
- `open` 時呼叫 `PlayerTransport.connect(...)`
- `message` 時呼叫 `PlayerTransport.receiveMessage(...)`
- `close` 時呼叫 `PlayerTransport.disconnect(...)`
- 不直接實作遊戲規則或遊戲狀態管理

### `PlayerApi`

- REST API
- 提供玩家身份建立與登入相關流程
- 提供 websocket 連線前需要的玩家與 session 資訊

## 關鍵判斷

### 為什麼不直接讓 `GameRunner` 依賴 `ws`

- `ws` 物件提供的是傳輸能力，不是遊戲領域語意
- 若 `GameRunner` 直接依賴 `ws`，會讓框架細節滲入應用層
- 後續若要替換 transport 實作、補測試、改變廣播策略，成本會變高

### 為什麼保留 `PlayerTransport`

- `PlayerTransport` 可以把 `playerId`、`gameId`、`connectionId` 這類語意集中管理
- `PlayerTransport` 可以把 websocket lifecycle 轉為應用層介面
- `GameRunner` 只需依賴 transport abstraction，而不需知道底層是 Elysia topic、ws send，或其他實作

### 為什麼拆出 `OnlineSessionRegistry`

- 若把在線玩家、對局關係、訊息送收、topic 訂閱都塞進 `PlayerTransport`，責任會過大
- 把在線狀態獨立出來後
  - `PlayerTransport` 偏向接線生
  - `OnlineSessionRegistry` 偏向狀態索引
  - `GameRunner` 偏向流程控制
- 這樣比較容易維護，也比較容易逐步演進

### 為什麼先不用 EventBus

- 目前主流程是單一路徑
  - 玩家訊息 -> `PlayerTransport` -> `GameRunner` -> `GameRule`
  - 狀態更新 -> `GameRunner` -> `PlayerTransport` -> websocket
- 這類流程用 direct call 最容易理解與除錯
- EventBus 比較適合以下情境
  - 一個事件需要多個模組同時訂閱
  - 要額外接通知、監控、成就、分析等旁支功能
  - 需要跨系統傳播事件
- 現階段先不引入，可保留未來迭代空間

## 建議通訊流程

### 玩家連線

1. 前端透過 `PlayerApi` 建立或取得玩家身份
2. 前端以 session 資訊建立 websocket 連線
3. `PlayerSocket` 驗證必要上下文後，呼叫 `PlayerTransport.connect(...)`
4. `PlayerTransport` 將連線註冊進 `OnlineSessionRegistry`

### 玩家加入遊戲

1. API 或 application service 更新 `Game` 狀態
2. 同步更新 `OnlineSessionRegistry` 中的 `playerId -> gameId`
3. `PlayerTransport` 依 registry 結果決定廣播目標

### 玩家行動

1. `PlayerSocket` 收到訊息
2. `PlayerSocket` 呼叫 `PlayerTransport.receiveMessage(...)`
3. `PlayerTransport` 將命令轉給 `GameRunner`
4. `GameRunner` 載入 `Game`
5. `GameRunner` 呼叫 `GameRule`
6. `GameRunner` 保存結果
7. `GameRunner` 呼叫 `PlayerTransport` 回送或廣播狀態

### 玩家離線

1. `PlayerSocket` 觸發 disconnect
2. `PlayerTransport` 清除該連線
3. `OnlineSessionRegistry` 更新在線狀態
4. 本輪不處理 reconnect 恢復策略

## 命名與範圍

### `SessionTransport` -> `PlayerTransport`

- 建議更名
- 理由是目前關心的不是 session 技術細節，而是玩家導向的通訊抽象

### `Match` -> `Game`

- 方向上可考慮
- 但這會影響資料模型、API 命名、儲存層與前端語意
- 本輪先不併入通訊架構調整，避免議題擴散

## 限制與考量

- 當前設計以單機程序內可維護性為主
- Elysia topic 可作為底層廣播優化，但不作為核心業務抽象
- `PlayerTransport` 不應同時兼任 repo、session 驗證、遊戲規則、狀態存放庫
- `GameRunner` 不應依賴 websocket 實體
- `OnlineSessionRegistry` 目前僅處理在線狀態，不處理 reconnect 恢復

## 目前結論

- 保留 `PlayerTransport` 抽象是合理的
- `SessionTransport` 應更名為 `PlayerTransport`
- `PlayerTransport` 應縮小成 transport adapter
- 在線玩家與遊戲關係應拆成 `OnlineSessionRegistry`
- 先採 direct call 設計，後續再視需要演進到 EventBus
- 單一玩家允許多連線
- reconnect 暫時不處理
