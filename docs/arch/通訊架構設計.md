# 通訊架構設計

## 目標

- 在個人可維護的範圍內，建立後端元件職責與通訊分層
- 保留 websocket 長連線模型
- 避免業務邏輯直接依賴 Elysia / Bun 的 `ws` 物件
- 先以容易理解與容易迭代的設計為主

## 設計原則

- `ws` 物件只作為 adapter 層資源，不直接成為業務模型
- 單一玩家允許多連線，方便單機 e2e 測試與多分頁操作

## 元件分層

### `GameRule`

- 遊戲規則
- 定義遊戲狀態轉移規則
- 提供純規則函式，例如 `move(playerId, action): GameState`
- 不依賴 websocket、repo、topic、Elysia 型別
- 應能獨立測試

### `Game`

- 特定一場遊戲的實例
- 包含遊戲狀態、玩家資訊、遊戲狀態欄位

### `GameRepo`

- 遊戲存放庫
- 負責讀寫 `Game`
- 提供 `get(gameId): Game` 與 `save(gameId, game)` 類型的介面

### `GameRunner`

- 遊戲執行器
- 作為應用層 orchestrator / command handler
- 接收玩家命令後
  - 載入 `Game`
  - 驗證玩家與遊戲上下文
  - 呼叫 `GameRule`
  - 保存新狀態
  - 要求 `PlayerTransport` 回送或廣播結果

### `PlayerTransport`

- 玩家通訊抽象
- 記載 ws 與 playerId 的對應關係
  - `Map<wsId, playerId>`
  - `Map<playerId, [ws]>`
- 配置 ws 訂閱 topic
  - playerId: `ws.subscribe(plays:playerId)`
  - gameId: `ws.subscribe(games:gameId)`
- 負責銜接 websocket 與應用層流程
  - `connect(ws, playerId)`
  - `receiveMessage(ws, message)`
  - `disconnect(ws)`
- 提供玩家導向的送收能力
  - `sendToPlayer(playerId, message)`
  - `receiveFromPlayer(handler)`
  - `sendToPlayersInGame(gameId, message)`
- 提供路由切換
  - `playerJoinGame(playerId, gameId)`
  - `playerLeaveGame(playerId, gameId)`

### `PlayerSocket`

- Elysia websocket adapter
- 只負責對接 websocket lifecycle
- `open` 時呼叫 `PlayerTransport.connect(...)`
- `message` 時呼叫 `PlayerTransport.receiveMessage(...)`
- `close` 時呼叫 `PlayerTransport.disconnect(...)`
- 不直接實作遊戲規則或遊戲狀態管理

### `PlayerApi`

- REST API
- 提供玩家身份建立與登入相關流程
- 提供 websocket 連線前需要的玩家與 session 資訊

## 通訊流程

### 玩家連線

1. 前端透過 `PlayerApi` 建立或取得玩家身份
2. 前端以 session 資訊建立 websocket 連線
3. `PlayerSocket` 驗證必要上下文後，呼叫 `PlayerTransport.connect(ws, playerId)`
4. `PlayerTransport` 將 ws 加入到 playerId - ws 的對應關係中
5. `PlayerTransport` 使用 `ws.subscribe(plays:playerId)` 訂閱玩家專屬頻道

### 玩家加入遊戲

1. API 或 application service 更新 `Game` 狀態
2. 同步呼叫 `PlayerTransport` 中的 `playerJoinGame(playerId, gameId)`
3. `PlayerTransport` 找到 `playerId` 對應的複數 `ws`，使用 `ws.subscripe(games:gameId)` 加入遊戲頻道

### 玩家行動

1. `PlayerSocket` 收到訊息
2. `PlayerSocket` 呼叫 `PlayerTransport.receiveMessage(ws, message)`
3. `PlayerTransport` 從 `ws` 找到 `playerId`
4. `PlayerTransport` 使用 `receiveFromPlayer` 中註冊的 handler 將 `playerId` 與 `message` 傳遞給應用層，例如 `GameRunner`
5. `GameRunner` 載入 `Game`
6. `GameRunner` 呼叫 `GameRule`
7. `GameRunner` 保存結果
8. `GameRunner` 呼叫 `PlayerTransport.sendToPlayersInGame(gameId, message)` 廣播結果

### 玩家離線

1. `PlayerSocket` 觸發 `PlayerTransport.disconnect(ws)`
2. `PlayerTransport` 從 `ws` 找到 `playerId`
3. `PlayerTransport` 從 playerId - ws 的對應關係中移除該 ws
